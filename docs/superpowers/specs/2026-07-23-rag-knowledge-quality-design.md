# RAG 产品资料质量完善设计

## 目标

将 PR 1 中 50 份“待人工审核”产品资料转化为可追溯、可分库、可自动验收的 RAG 数据，同时避免未经核实的参数进入客服事实回答。

## 数据分库

- **小米生态核心库**：30 份小米、REDMI、米家资料，用于产品知识问答。
- **竞品选购对比库**：20 份 iQOO、OPPO、vivo、华为、荣耀、联想拯救者、红魔资料，只用于选购和对比场景。
- 两个知识库使用独立 collection，不允许普通产品客服问答跨库召回竞品资料。

## 文档质量契约

每份 Markdown 必须包含统一元数据：文档编号、标题、类别、品牌、型号、发布日期、语言、整理日期、审核状态、主要来源 URL、来源类型、来源等级和价格来源 URL。

可检索正文不得包含“待复核、待确认、待人工打开原文”等未完成表述。无法从来源确认的参数不进入事实正文，统一移入文末“审核备注”，且审核状态保持 `需人工复核`；只有来源明确支持的内容才标记为 `可入库`。

来源 URL 从现有审核表回填。资料仍区分官方来源和专业媒体来源，回答与引用不得把媒体实测描述成官方标称。

## 接入与实体识别

- 批量入库脚本按品牌自动选择核心库或对比库。
- 上传时将文档的主要来源 URL 传入现有 `source_url` 字段，使来源覆盖率达到 100%。
- Markdown 解析器读取 front matter 中的品牌与型号，作为通用产品实体；现有正则提取作为兼容回退。
- 知识图谱和分析接口应识别全部 50 个文档对应的产品，而不是当前的 16 个。

## 自动质量门禁

新增可重复运行的校验脚本与测试，至少验证：

1. 文档总数为 50，核心库 30、对比库 20。
2. 文档编号、文件名和型号唯一。
3. 必填元数据完整且 URL 可解析。
4. 可入库正文不存在待复核标记。
5. 所有文档均能解析、切分并提取目标型号。
6. 临时 SQLite/Chroma 入库 50/50 成功、来源覆盖率 100%、产品实体数 50。
7. 小米问答只引用核心库；竞品问题只在对比库中回答。

## 审核表

保留原 Excel 样式并更新人工审核状态、审核备注和最终分库列。公式必须可重新计算且无 Excel 错误。审核表仍是来源台账，Markdown 是运行时知识载体。

## Docker 合并门槛

只有在质量门禁、后端全量测试和临时 RAG 回归全部通过后，才将整理后的资料合并到本地主代码并重建 Docker。现有 Docker 数据先备份；新建两个知识库后导入资料，不覆盖原知识库。前端使用队长版本并展示两个知识库供选择。

## 安全边界

- 不使用 DeepSeek 批量生成或“补全”产品事实。
- 不把自动整理等同于人工核验。
- 价格、上市时间和地区版本均保留数据日期与适用范围。
- 无可靠来源或仍需人工复核的参数继续触发兜底，不进入正式回答。
